## 1. 项目概述

### 1.1 项目定位

本项目是一个基于 [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) 的天气查询 Server，面向 AI 客户端（如 Cursor、Claude Desktop）提供结构化天气工具能力。数据源采用[高德地图天气查询 API](https://lbs.amap.com/api/webservice/guide/api/weatherinfo/)，以 npm 包形式发布，支持通过 `npx` 零安装启动。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| 协议标准化 | 遵循 MCP 规范，与主流 AI 客户端无缝集成 |
| 开箱即用 | 发布到 npm 后，用户仅需配置 API Key 即可使用 |
| 轻量可维护 | 单文件入口、依赖少、构建产物清晰 |
| 类型安全 | 全链路 TypeScript，参数与响应均有类型约束 |

### 1.3 系统架构

```
┌─────────────────┐     stdio (JSON-RPC)     ┌──────────────────────┐
│  MCP Client     │ ◄──────────────────────► │  xjt-weather-mcp     │
│  (Cursor 等)    │                          │  (Node.js Server)    │
└─────────────────┘                          └──────────┬───────────┘
                                                        │ HTTPS
                                                        ▼
                                             ┌──────────────────────┐
                                             │  高德天气 Web 服务    │
                                             │  weatherInfo API     │
                                             └──────────────────────┘
```

**通信方式**：MCP Server 通过 `StdioServerTransport` 与 Client 交换 JSON-RPC 消息；日志输出至 `stderr`，`stdout` 专用于协议通信。

**暴露工具**：

| Tool | 功能 |
|------|------|
| `get-weather-live` | 查询指定城市（6 位 adcode）实况天气 |
| `get-weather-forecast` | 查询指定城市当天及未来 3 天预报 |

## 3. 运行时与语言

### 3.1 Node.js ≥ 18

**选型理由**：

- MCP 官方 TypeScript SDK 面向 Node.js 生态，示例与文档均以 Node 为主。
- Node 18+ 内置原生 `fetch`，无需额外 HTTP 客户端依赖，可直接调用高德 REST API。
- 支持 ES Module（`"type": "module"`），与 MCP SDK 的 ESM 导出一致。
- 与 `npx` 分发模式天然契合，用户无需预装运行时以外的依赖。

**未选方案**：

| 方案 | 未选原因 |
|------|----------|
| Python + FastMCP | 团队 TypeScript 栈统一；npm 分发对前端/Node 开发者更友好 |
| Deno / Bun | MCP 生态与示例以 Node 为主，兼容性风险较高 |
| 独立 HTTP 服务 | MCP stdio 模式更轻量，无需端口管理与进程守护 |

### 3.2 TypeScript 5.9

**选型理由**：

- 编译期类型检查，减少 MCP 工具参数与高德 API 响应的结构错误。
- `module: NodeNext` + `moduleResolution: NodeNext` 与 Node ESM 规范对齐。
- `strict: true` 开启严格模式，提升代码可靠性。
- 独立 `tsconfig.scripts.json` 覆盖测试脚本，与 Server 源码分离。

**配置要点**：

```json
{
  "target": "ES2022",
  "module": "NodeNext",
  "strict": true,
  "noEmit": true
}
```

类型检查由 `tsc --noEmit` 负责，实际产物由 Vite 构建生成。

## 4. MCP 协议与 SDK

### 4.1 @modelcontextprotocol/sdk

**选型理由**：

- MCP 官方维护的 TypeScript SDK，协议版本与规范保持同步。
- 提供 `McpServer`、`StdioServerTransport` 等高层抽象，开发成本低。
- 内置 Zod 集成（`server.tool()`），工具 schema 可自动生成 Client 侧描述。
- 项目参考官方 [Weather Server Quickstart](https://github.com/modelcontextprotocol/typescript-sdk/blob/main/examples/server-quickstart/src/index.ts) 实现，降低协议理解成本。

**使用方式**：

```typescript
const server = new McpServer({ name: 'xjt-weather-mcp', version: PACKAGE_VERSION });
const transport = new StdioServerTransport();
await server.connect(transport);
```

### 4.2 传输层：stdio

**选型理由**：

- MCP 推荐的标准传输方式之一，Client 通过子进程 stdin/stdout 通信。
- 无需网络端口，适合本地 AI 工具集成场景。
- 配置简单：`command` + `args` + `env` 即可在 Cursor / Claude 中启用。

**未选方案**：

| 方案 | 未选原因 |
|------|----------|
| SSE / HTTP | 需独立部署与鉴权，超出 Quickstart 轻量定位 |
| WebSocket | 同上，运维复杂度更高 |

## 5. 参数校验：Zod 4

**选型理由**：

- MCP SDK 原生支持 Zod schema 作为工具参数定义，自动生成 JSON Schema。
- 城市编码（adcode）使用正则约束，无效输入在调用前即被拦截：

```typescript
const citySchema = z
  .string()
  .regex(/^\d{6}$/)
  .describe('城市 adcode，6 位数字（例如 110101 北京东城，310100 上海）');
```

- Zod 4 与 SDK 版本配套，类型推断体验更好。

**未选方案**：

| 方案 | 未选原因 |
|------|----------|
| 手写校验 | 重复劳动，无法自动生成 Client 侧 schema |
| JSON Schema 直写 | 与 SDK 集成度低，维护成本高 |
| Joi / Yup | MCP SDK 官方示例与生态以 Zod 为主 |

## 6. 数据源：高德地图天气 API

### 6.1 选型理由

| 维度 | 说明 |
|------|------|
| 国内覆盖 | 中国大陆城市 adcode 体系统一，文档与编码表完善 |
| 接口简洁 | 单一 REST 端点，通过 `extensions=base|all` 区分实况/预报 |
| 免费额度 | Web 服务 Key 对个人/小规模调用足够 |
| 响应结构稳定 | `status` + `infocode` 标准错误码，易于封装 |

### 6.2 接口设计

- **端点**：`https://restapi.amap.com/v3/weather/weatherInfo`
- **鉴权**：环境变量 `AMAP_API_KEY`（Web 服务类型 Key）
- **参数**：`city`（6 位 adcode）、`extensions`（`base` 实况 / `all` 预报）、`output=JSON`

### 6.3 未选方案

| 方案 | 未选原因 |
|------|----------|
| OpenWeatherMap | 国内访问与中文城市名支持较弱 |
| 和风天气 | 亦可考虑，本项目以高德 Quickstart 场景为主 |
| 自建爬虫 | 合规与稳定性风险高 |

### 6.4 HTTP 客户端

使用 Node 18+ 原生 `fetch`，无 `axios` / `node-fetch` 等额外依赖，减少包体积与供应链风险。

## 7. 构建与打包：Vite 6 (SSR Lib 模式)

### 7.1 选型理由

- **单文件 ESM 输出**：`dist/server.js` 作为 bin 入口，结构清晰。
- **依赖外置**：`@modelcontextprotocol/sdk`、`zod` 保留为 `dependencies`，由 `npx` 安装时拉取，避免重复打包。
- **Shebang 注入**：Rollup `banner: '#!/usr/bin/env node'`，可直接作为 CLI 执行。
- **开发体验**：`vite build -w` 支持 watch 模式，改代码后自动重建。
- **构建速度**：Vite 基于 Rollup，对小体量 Node 项目构建足够快。

### 7.2 构建配置要点

```typescript
const dependencies = Object.keys(pkg.dependencies ?? {});

rollupOptions: {
  external: (id) =>
    id.startsWith('node:') ||
    dependencies.some(
      (dependency) => id === dependency || id.startsWith(`${dependency}/`)
    ),
}
```

### 7.3 为什么 external `dependencies`？

Vite/Rollup 默认会把 `import` 的模块**打进**产物。对本项目这类 **npm CLI 包**，更合适的策略是：**只打包自己的业务代码，把 `package.json` 里的 `dependencies` 留到运行时由 Node 从 `node_modules` 加载**。

#### 7.3.1 构建时发生了什么

```
源码 server.ts
  ├── import from 'node:module'          → external（Node 内置）
  ├── import from '@modelcontextprotocol/sdk/...'  → external（dependencies）
  ├── import from 'zod'                  → external（dependencies）
  └── 业务逻辑（fetch、格式化、tool 注册）  → 打进 dist/server.js
```

当前产物约 **5 KB**，其中保留的是 `import` 语句，例如：

```javascript
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import { z } from 'zod';
```

若不做 external、全部 bundle 进 `dist/server.js`，体积会膨胀到数百 KB 甚至更大（MCP SDK + Zod 及其传递依赖），且与 npm 安装机制重复。

#### 7.3.2 核心原因

| 原因 | 说明 |
|------|------|
| **符合 npm 包语义** | `dependencies` 的设计就是「安装包时一并安装、运行时 `require`/`import` 解析」。`npx @xjt-demo/xjt-weather-mcp` 会先装本包及其依赖，再执行 `dist/server.js`，Node 自然能从 `node_modules` 找到 SDK 和 Zod。 |
| **减小发布体积** | 发到 npm 的 `files` 只有 `dist/`（约 5 KB）+ 文档。SDK 不打进 tarball， tarball 更小，下载更快。 |
| **避免重复与版本冲突** | 若 bundle 了 SDK，同时又把 SDK 写在 `dependencies` 里，等于同一份代码存在两份；若用户环境里还有其他 MCP 相关包，还可能引发重复实例或版本不一致问题。 |
| **依赖升级路径清晰** | 安全补丁或 minor 升级可通过 `pnpm update` / npm 正常的 semver 范围（`^1.21.3`）流转，无需每次重新 bundle 发版。 |
| **Node 内置模块必须 external** | `node:module`、`node:path` 等由运行时提供，不能也不应打进 bundle。 |
| **SDK 子路径导出更稳定** | MCP SDK 使用 `@modelcontextprotocol/sdk/server/mcp.js` 等子路径导出；保持 external 后，由 Node ESM 解析器按 package exports 加载，比 Rollup 强行打包更不容易踩坑。 |

#### 7.3.3 运行时解析流程（npx 场景）

```
npx -y @xjt-demo/xjt-weather-mcp
        │
        ▼
npm 安装包到缓存目录
  ├── dist/server.js          ← bin 入口（仅业务代码）
  └── node_modules/
        ├── @modelcontextprotocol/sdk/
        └── zod/
        │
        ▼
node dist/server.js
  └── ESM import 解析 → 从 node_modules 加载 SDK、Zod
```

这与「把 SDK 打进 dist、发布一个巨型单文件」不同：后者适合 **standalone 可执行文件**（如 `pkg`、`nexe`），不适合 **标准 npm 包 + npx** 的分发模型。

#### 7.3.4 为什么从 `package.json` 动态读取

```typescript
const dependencies = Object.keys(pkg.dependencies ?? {});
```

这样新增 `dependencies` 时**不必改 Vite 配置**，构建会自动 external 新依赖。规则是：

- **`dependencies`** → external（运行时必需，随包安装）
- **`devDependencies`** → 不 external，但也不应被 `server.ts` 引用（如 `vite`、`tsx` 仅开发/测试用）
- **`node:*`** → 始终 external

#### 7.3.5 若不做 external 会怎样

| 问题 | 后果 |
|------|------|
| 产物体积暴涨 | `dist/server.js` 从 ~5 KB 变为数百 KB+ |
| 发布包冗余 | tarball 里既有 bundle 里的 SDK，又有 `node_modules` 里的 SDK |
| 调试困难 | 堆栈指向 bundle 内联代码，而非 node_modules 源文件 |
| 潜在兼容性风险 | SDK 内部若依赖 Node 特有行为或动态加载，bundle 后可能行为异常 |

#### 7.3.6 何时才应该 bundle 依赖

以下场景才考虑把依赖打进产物：

- 发布 **零依赖单文件**（不依赖 npm install 第三方包）
- 目标环境 **没有** `node_modules`（浏览器、Edge Worker 等）
- 需要 **完全锁定** 某一版本依赖、不接受用户侧 semver 解析

本项目通过 `npx` 分发、依赖 npm 安装机制，因此 **external dependencies 是正确默认策略**。

**未选方案**：

| 方案 | 未选原因 |
|------|----------|
| tsc 直出 | 需额外处理 shebang、路径别名与产物结构 |
| esbuild 裸用 | 配置分散，不如 Vite lib 模式一体化 |
| Webpack | 对小项目过重，冷启动与配置成本高 |
| tsx 直跑源码 | 不适合 npm 发布场景，用户需完整 devDependencies |

## 8. 包管理与分发

### 8.1 pnpm

**选型理由**：

- `packageManager: pnpm@10.22.0` 锁定版本，保证团队与 CI 环境一致。
- 磁盘效率高，依赖安装速度快。
- 与 npm 发布流程兼容（`publishConfig.access: public`）。

### 8.2 npm + npx 分发

**选型理由**：

- MCP 客户端配置仅需 `npx -y @xjt-demo/xjt-weather-mcp`，零全局安装。
- `bin` 字段指向 `dist/server.js`，符合 CLI 包惯例。
- `files: ["dist", "README.md", "LICENSE"]` 精简发布体积。
- `prepublishOnly` 钩子确保发布前自动构建。

## 10. 发布与版本管理

### 10.1 release-it

**选型理由**：

- 一条命令完成：版本号 bump → CHANGELOG 更新 → Git tag → npm publish → GitHub Release。
- `@release-it/keep-a-changelog` 插件自动维护 `CHANGELOG.md`。
- `before:init` 钩子强制执行 `typecheck` + `build`，防止坏版本流出。

### 10.2 分支与发布通道

采用 **main（稳定）+ alpha（预发布）** 双分支策略，分支与 npm dist-tag 一一对应：

| 分支 | 命令 | npm dist-tag | 版本示例 | 用途 |
|------|------|--------------|----------|------|
| `alpha` | `pnpm release:alpha` | `alpha` | `1.2.0-alpha.0` | 新功能内测、预发布验证 |
| `main` / `master` | `pnpm release:stable` | `latest` | `1.2.0` | 稳定正式版 |

配置文件：

- `.release-it.alpha.json`：`requireBranch: ["alpha"]`，GitHub Release 标记为 prerelease
- `.release-it.json`：`requireBranch: ["main", "master"]`，发布到 `latest`

典型流程：在 `alpha` 分支迭代并发布 `-alpha.N` → 验证通过后 merge 到 `main` → 在 `main` 上执行 `release:stable` 发布同版本号的正式版（如 `1.2.0`）。

## 11. 安全与配置

| 项 | 策略 |
|----|------|

## 14. 后续演进建议

以下为可选扩展方向，当前版本未实现：

1. **缓存层**：对同一 adcode 短时缓存，降低高德 API 调用频率。
2. **城市名解析**：增加「城市名 → adcode」工具，提升 LLM 调用体验。
3. **SSE 传输**：若需远程部署，可扩展 HTTP/SSE 传输模式。
4. **Vitest 单元测试**：对格式化函数（`formatLiveWeather` 等）补充纯函数测试。
5. **CI 流水线**：GitHub Actions 自动 typecheck、build、发布前冒烟测试。
